# TFDA 醫材查驗登記智慧審核系統：全面技術規格與法規判定邏輯白皮書 (V2.1)

## 壹、 系統總覽與開發背景 (Project Overview & Background)

### 1.1 數位轉型下的法規挑戰
在台灣醫療器材管理法 (Medical Device Act) 正式施行後，醫材從藥事法中獨立出來，其審核機制變得更加精細。對於法規事務人員 (RA) 而言，面對成百上千種不同用途、風險、產地的醫材，如何精確判斷「應送審文件清單」並確保數千頁的「技術文件 (STED)」內容一致，已成為企業效率的瓶頸。

本系統「TFDA 醫材查驗登記智慧審核系統」旨在構建一個數位化的法規大腦。其核心價值在於將非結構化的法規文本轉化為結構化的邏輯代碼，輔以最新的大型語言模型 (LLM)，實現由「資料整理」到「智慧審計」的跨越。

### 1.2 系統核心功能
- **動態法規路徑引擎 (Strategic Path Engine)**: 多維度判定送審清單。
- **文件智能審計 (Interrogation Engine)**: 透過雙代理人 (Dual-Agent) 執行深度文件交叉比對。
- **本地 PDF 實體處理 (Local PDF Processing)**: 瀏覽器端切片與安全審查。
- **規則庫編輯模組 (Logic Management)**: 自定義 mdpass 與 conflict 規則。

## 貳、 系統架構與技術細節 (System Architecture)

### 2.1 前端技術矩陣 (Frontend Stack)
系統基於 **Vite + React 18** 構建，這是一個對效能與開發速度最優化的組合。
- **TypeScript**: 強型別確保了法規邏輯在代碼層級的零錯誤率。特別是在處理 `Scenario` 與 `Rule` 介面時，嚴謹的定義避免了邏輯崩潰。
- **Tailwind CSS + Neo-Brutalist Design**: UI 設計不盲目追求美觀，而是以「法規嚴謹」為核心。使用大量的網格、邊框與單色調，讓法規人員能專注於內容，同時利用 `accent-red` 標記出潛在的合規風險。
- **Framer Motion (Motion)**: 本系統採用 Motion 進行高度自定義的視覺反饋。例如：在執行 AI 審查時，會有「掃描線矩陣」特效，這不僅是裝飾，更是為了緩解用戶在等待大規模模型推理時的視覺壓力。

### 2.2 核心文件處理邏輯：pdf-lib 之應用
與傳統將檔案上傳至後端解析的方案不同，本系統採用 **完全客戶端處理解析 (Full Client-side Processing)**。
1.  **二進制讀取**: 檔案透過 `FileReader` 以 `ArrayBuffer` 讀入。
2.  **PDF 實體化**: 使用 `pdf-lib` 的 `PDFDocument.load` 將檔案載入記憶體。
3.  **頁面切片 (Page Trimming)**: 
    - 法規文件的技術章節往往位於特定區間（例如臨床數據可能在 15-40 頁）。
    - 系統允許用戶輸入特定號範圍，如 `1-5, 20, 35-40`。
    - 技術實現：系統調用 `copyPages` 方法從原始文檔拷貝指定頁面，並通過 `addPage` 插入新文檔。最後導出為 Base64 字串送交 AI。
    - **優勢**: 大幅減少 Context Tokens。Gemini 雖然具備長文本能力，但精細的 Context 通常能獲得更高的邏輯推理準確度。

## 參、 詳細法規判定邏輯 (Detailed Regulatory Logic)

本章節詳述系統如何根據用戶輸入的參數設定，自動計算出 TFDA 要求的「醫療器材查驗登記申請書附件清單」。

### 3.1 判定因子詳解
系統判定邏輯是基於一個多維張量 (Multidimensional Tensor)：

#### A. 產品分級 (Classification)
- **第一等級 (Class I)**: 低風險醫材。
    - 主路徑：僅需提供基本的行政文件。
    - 備註：若為無菌第一級，則需額外檢附滅菌證明。
- **第二、三等級 (Class II/III)**: 中高風險醫材。
    - 主路徑：需檢附「STED 技術文件摘要」。
    - 臨床資料要求：第三等級通常強制要求臨床評估報告 (CER) 或臨床試驗報告。

#### B. 產地與來源 (Origin & Source)
- **國產醫材 (Domestic)**: 
    - 核心文件：工廠登記證、製造業醫材商許可證、品質系統文件 (QSD)。
- **輸入醫材 (Import)**: 
    - 核心文件：出產國許可證 (CFS)、委託代理授權書 (LoA)、國外製造廠品質系統文件 (QSD)。
    - 特定國家邏輯：若產地為中國大陸，需額外檢核特定形式的批准證明文件。

#### C. 特定屬性篩選器 (Attribute Filters)
- **滅菌產品 (Sterile)**: 必須包含滅菌確認報告 (D8) 與包裝完整性測試。
- **含軟體產品 (SaMD)**: 必須包含軟體版本說明、網路安全 (Cybersecurity) 說明。
- **包含動物來源 (Animal Tissue)**: 需檢附病毒滅活確效報告。

### 3.2 文件編號映射系統 (Document Code Mapping)
系統預設了 TFDA 標準的 D 系列代碼：
- **D1**: 醫材商許可證影本
- **D2**: 切結書（甲/乙/丙）
- **D3**: 國外製造廠品質管理系統證明文件 (QSD)
- **D4**: 出產國許可證 (CFS)
- **D5**: 委託代理授權書 (LoA)
- **D6**: 說明書 (IFU)、標籤、包裝、仿單草案
- **D7**: 產品結構、規格、性能、用途、圖樣等技術資料 (STED)
- **D8**: 安全及效能基本要求 (Essential Principles)
- **D9**: 風險管理報告 (ISO 14971)

### 3.3 邏輯衝突檢核 (Conflict Rules Logic)
除了「需要哪些文件」外，系統還具備「邏輯矛盾檢測」。
例如：
- 如果 `Selection.Origin == 'IMPORT'` 且 `D4_exists == false` -> 系統在 Inspector 模組中會彈出「嚴重警告：輸入產品缺少 CFS」。
- 如果 `Selection.Attributes.isSoftware == true` 且 `D7` 中未提及軟體驗證 -> 觸發「AI 建議：請確認技術文件中是否已包含軟體描述章節」。

## 肆、 AI 多代理人審核模型 (Dual-Agent Collaboration Model)

### 4.1 Agent 1: 策略架構師 (Regulatory Strategist)
- **核心 Prompt**: 「你是一位擁有 20 年經驗的 TFDA 資深法規顧問。請分析這份技術文檔，針對產品名稱、分類品目碼、預期用途進行一致性審查。」
- **任務**: 確保產品的 Claim (宣稱) 與技術規格相符。

### 4.2 Agent 2: 合規審計師 (Compliance Auditor)
- **核心 Prompt**: 「你是一位細心的合規審計員。請針對文件中的數據（如 API, 滅菌劑殘留量, 電性安全測試數據）進行交叉驗證。檢查標籤上的資訊是否與技術報告一致。」
- **任務**: 挖掘文件中的微觀錯誤。

### 4.3 合併輸出與結果修正
系統會將兩個 Agent 的輸出彙整。用戶可以切換「編輯模式」，直接在 Markdown 視圖中修改 Agent 的錯誤判斷。這體現了 **Human-in-the-loop (人工在環)** 的重要性，因為法規決策最終仍需由法規人員背書。

## 伍、 本地化規則庫管理說明 (Rules Selection & Modification)

### 5.1 mdpass_rules.json (路徑清單規則)
此文件控制系統的「場景判定」。
- **更新邏輯**: 當 TFDA 公告某類醫材（如 Class II 某些品目）改為簡化審查時，管理員可導出此 JSON，修改對應的 `req` 陣列（移除特定 D 代碼），再重新上傳。
- **實踐**: 確保所有業務邏輯與代碼分離，實現熱更新。

### 5.2 advanced_conflict_rules.json (審核邏輯規則)
此文件控制「智慧警示」。
- **表達式支援**: 支持布林運算與變量標記。
- **目的**: 讓大企業能定義自己的「內部審核標準」（如：所有外銷產品必須包含特定語言標籤），而不僅僅是满足 TFDA 的最低要求。

## 陸、 總結與前瞻 (Summary & Future Outlook)
本系統不僅是一個查驗登記的工具，它是一個「法規智庫的載體」。透過將複雜的判定邏輯轉化為可視化的數位流程，我們大幅度降低了法規進入壁壘，提升了台灣醫療器材產業的競爭力。

未來，我們將計畫導入：
1.  **動態 OCR 技術**: 直接從掃描版 PDF 中提取 Table 數據進行對齊。
2.  **法規異動推播**: 連結衛福部 API，當法規變動時，自動在系統內標記受影響的舊案。

## 柒、 二十個深度追蹤與法規 Follow-up 問題 (Total 20 Qs)

1.  **歐盟 MDR vs. TFDA STED**: 當企業已經準備好 EU MDR 技術文件時，系統如何最快地映射到 TFDA STED 的 D7 附件中？特別是關於 Post-Market Surveillance (PMS) 部分的差異判定？
2.  **臨床證據等同性 (Clinical Equivalence)**: 在 D10 判定邏輯中，如何定義「與類似品具備等同性」？Agent 是否能根據說明書自動判斷兩者之預期用途 (Intended Use) 是否完全一致？
3.  **無菌級別與包裝驗證**: 針對無菌產品，系統如何判定是否需要包含「五年的安定性試驗 (Stability Test)」？當包裝材質變更時，規則庫是否能自動偵測到需要補做滅菌確效？
4.  **軟體變更申報策略**: 在 SaMD (醫療器材軟體) 模組中，Agent 如何區分「重大變更 (Major Change)」與「微幅變更 (Minor Change)」？這將決定是否需重新進行查驗登記。
5.  **跨國製造場品質證明 (QSD) 之有效期**: 系統如何檢查 QSD 的工廠認證日期？並在過期前三個月自動觸發警示？
6.  **中國大陸生產醫材之特殊性**: D4 (許可證) 的形式要求與歐美不同。系統如何檢查其「醫療器材註冊證」中的內容是否與 D6 (中文仿單) 的品名完全一致？
7.  **標籤與仿單 (D6) 之核定邏輯**: TFDA 對於標籤字體大小、警告圖示有明確要求，Agent 能否視覺化分析標籤 PDF，判定圖記是否符合 CNS 標準？
8.  **風險管理報告 (ISO 14971) 之深度審計**: Agent 如何判斷風險分析矩陣中是否有「遺漏的危害 (Hazards)」？例如電性醫材未考慮到軟體失效引起的熱傷害。
9.  **生物相容性之豁免條件**: 針對非接觸人體 (Non-contacting) 或短暫接觸 (Transient) 醫材，系統如何透過法規邏輯自動生成「豁免說明書」草稿？
10. **藥物醫材複合產品 (Combination Products)**: 判定邏輯如何切換。當產品被判定為「以藥物為主」時，系統應如何導向至藥政審查路徑？
11. **TFDA 技術審查 (Technical Review) 缺失函 (Deficiency Letter) 預測**: 系統是否能根據過往被退件的案例，分析當前文件最可能被補件的熱點在哪裡？
12. **電子仿單 (e-IFU) 代碼對齊**: 針對醫院用儀器，系統如何導引用戶正確套用 TFDA 的電子仿單條款並在標籤上標示讀取連結？
13. **類似品 (Similar Product) 目錄管理**: 系統是否能維護一個企業內部類似品庫，以便在不同申請案中快速複用性能比對數據？
14. **雙代理人 (Agent1/Agent2) 之衝突解決**: 當兩個 Agent 對同一份文件的合規性持有相反意見時，系統如何記錄這種「不確定性」並引導用戶裁決？
15. **臨床試驗 (Clinical Trial) 豁免判定**: 針對新型醫材，系統如何根據「是否具備全新原理」自動判定是否必須留台執行臨床試驗？
16. **製造商與代理商之 LoA 有效期管理**: 針對進口商，系統如何自動檢核授權書 (LoA) 的授權範圍是否包含「查驗登記」與「產品售後追蹤」？
17. **體外診斷試劑 (IVD) 之特定邏輯**: 針對 IVD，判定清單中如何自動切換為「CoA 批次檢驗報告」而非一般的 STED？
18. **UDI (唯一識別碼) 之標示實作**: 系統如何核對 D6 文件中的 DI (Device Identifier) 與 PI (Production Identifier) 佔位符是否正確？
19. **Real-World Evidence (RWD/RWE) 之採納度分析**: 當臨床數據不足時，Agent 如何評估該產品提供的真實世界數據是否足以支撐其臨床宣稱？
20. **綠色通道與突破性醫材 (Breakthrough Device)**: 系統如何判定該醫材符合「極重度傷病」或「國內臨床急需」，進而啟動優先審查路徑？
