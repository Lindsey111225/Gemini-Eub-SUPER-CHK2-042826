# TFDA 醫療器材查驗登記智慧審核系統：全面技術規格與法規判定邏輯白皮書 (V2.1)

## 壹、 導言：數位法規時代的合規轉型 (Introduction)

### 1.1 開發動機與行業痛點
在當前全球醫療器材法規收嚴的趨勢下，台灣醫療器材管理法 (Medical Device Act) 的施行將醫材管理從藥事法中獨立出來，其審核廣度與深度皆大幅提升。對於法規事務人員 (Regulatory Affairs, RA) 而言，面對衛福部食品藥物管理署 (TFDA) 複雜的查驗登記準則，往往面臨「決策路徑不明」、「文件一致性難以查檢」以及「人工審核效率低下」等核心痛點。

本系統——「TFDA 醫療器材查驗登記智慧審核系統」——旨在將非結構化的法規指南轉化為結構化的運算邏輯。透過「動態路徑判定引擎」與「Google Gemini 1.5 驅動的多代理人 AI 審計系統」，我們為醫材企業提供了一個全方位的智慧合規工作平台。

### 1.2 系統願景
我們的願景是構建一個「法規智庫」，讓企業在產品研發初期即可精確預測查驗登記所需的成本與時間，並在送審前模擬官方審核流程，顯著降低補件率 (Deficiency Rate) 與退件率。

## 貳、 技術架構與模組化開發 (Technical Stack & Architecture)

### 2.1 底層框架與型別規範
本系統採用 **Vite + React 18 + TypeScript** 的黃金技術棧。
- **Vite**: 提供極速的熱更新 (HMR)，對於需要頻繁調試法規規則庫的前端開發至關重要。
- **TypeScript**: 為了確保法規邏輯的嚴密性，系統內的 `Selection`, `Attribute`, `Scenario`, `Rule` 等核心數據實體皆有嚴格的介面 (Interface) 定義。這有效地防止了在處理複雜 JSON 規則庫時發生的型別混淆。

### 2.2 UI 設計哲學：新大膽設計 (Neo-Brutalism)
醫材法規是一門極其嚴肅的科學。因此，UI 設計摒棄了花哨的漸層與柔和的陰影，轉而採用高對比度的「瑞士美學網格系統」：
- **邊框線條**: 粗壯的 2px 實線邊框，強調區塊感。
- **色彩語彙**: 採用純白底色配以「醒目紅 (Accent Red)」作為強調色，用於標記潛在的合規風險與核心行動按鈕。
- **空間節奏**: 透過精確的內距 (Padding) 設定，模擬正式法規文件的閱讀層級，提升法規人員的工作專注度。

### 2.3 核心元件整合
- **Motion (Framer Motion)**: 系統整合了 `motion/react` 實作複雜的 AI 運算階段視覺反饋。例如在「Interrogation」階段，系統會模擬數值掃描與矩陣跳動效應，這不僅提升了用戶體驗，也直觀地顯示了後端大型語言模型的推理活躍度。
- **pdf-lib (客戶端實體處理)**: 這是本系統的核心技術優勢之一。為了確保醫材機密文件的隱私安全性（Privacy-by-Design），系統利用 `pdf-lib` 在用戶端瀏覽器中直接進行 PDF 的解析、渲染與「頁面切片」。
    - **智慧切片技術**: 法規文件通常含有大量冗長附件。RA 用戶可以使用「Trim Range」功能，指定如 `1-5, 20-30` 頁。系統會在本地暫存區生成一個新的封裝二進制流，僅將精簡後的文本或圖像數據發送至 AI API，大幅優化了 Token 消耗與推理品質。

## 參、 詳細法規清單判定邏輯 (Document List Generation Logic)

系統的核心靈魂在於其「全自動法規路徑判定引擎」。此模組模擬了 TFDA 醫材分類判定與查驗登記附件要求的完整決策路徑。

### 3.1 判定變量定義 (Input Dimensions)
系統透過以下五個維度進行聯集運算，決定最終的必備文件清單 (Required Document List)：

1.  **Risk Level (風險分級)**: 
    - **第一等級 (Class I)**: 通常對應簡易路徑，重點在於行政文件的完整性。
    - **第二、三等級 (Class II & III)**: 對應 STED (Summary Technical Documentation) 結構化申報，需要深度的技術效能評估報告。
2.  **Origin (產地)**: 
    - **及國產 (Domestic)**: 查驗重點在於製造業藥商許可證與 GMP/QMS 之對應性。
    - **輸入 (Import)**: 觸發出產國許可證 (CFS) 與授權書 (LoA) 的硬性檢核。同時需對產地 (Origin Country) 為中國大陸的產品啟動特定證照檢測邏輯。
3.  **Path Strategy (申報路徑)**: 
    - **基本登記 (Basic)**: 針對全新或未有類似品之產品。
    - **類似品比對 (Similar)**: 適用於二、三級醫材判斷其性能是否與已核准之醫材等同。
4.  **Device Attributes (產品關鍵屬性)**:
    - **Sterile (無菌)**: 是否具備滅菌包裝與確效報告 (D8)。
    - **Software/SaMD (軟體)**: 是否需檢附軟體版本與網路安全 (Cybersecurity) 說明。
    - **Radiation (輻射)**: 是否涉及電離或非電離輻射之安全性測試。
    - **China Origin (大陸產地)**: 是否符合《大陸地區醫材輸入特殊規定》。

### 3.2 文件代碼系統 (Document Code Index)
系統預設了 TFDA 標準的 D 系列代碼，並與規則庫 (`mdpass_rules.json`) 高度對齊：
- **D1**: 醫療器材查驗登記申請書及各式行政證明文件。
- **D2**: 切結書（甲、乙或丙）。
- **D3**: 國外製造廠品質管理系統證明文件 (QSD/GMP)。
- **D4**: 出產國最高衛生主管機關出具之產品販賣許可證 (CFS)。
- **D5**: 原廠委託代理授權書 (LoA)。
- **D6**: 產品結構、規格、性能、用途、圖樣、標籤、仿單、外盒等技術資料。
- **D7**: 產品之技術文件 (STED) 或相關檢驗報告。
- **D8**: 品質及其效能符合要求之證明文件 (Essential Principles)。
- **D9**: 風險管理報告 (ISO 14971)。

### 3.3 邏輯演算過程 (Algorithm Breakdown)
判定引擎採用「場景匹配 (Scenario Matching)」算法：
1.  **特徵提取**: 系統收集所有 Selection 狀態。
2.  **規則過濾**: 檢索 `mdpass_rules.json` 中的 `scenarios`。
3.  **匹配評分**: 當 Scenario 中的 `conditions` 完全符合用戶選取之特徵時，該路徑被激活。
4.  **矩陣疊加**: 系統會掃描所有匹配的路徑，將其 `req` (Required Documents codes) 數組合併，並去重。
5.  **警示生成**: 同時掃描 `advanced_conflict_rules.json`，若某兩項選擇存在邏輯互斥（如：產地為輸入但無 LoA 預期），系統會顯示「邏輯不一致警告」。

## 肆、 智慧審訊引擎與 AI 多代理人技術細節 (Interrogation Engine)

### 4.1 雙代理人審核模型 (Agent-based Collaboration)
為了模擬真實審查環境中的「多重審計」機制，我們設計了兩個職能完全不同的 AI 代理人：

#### 代理人一：法規戰略家 (Agent 1: Regulatory Strategist)
- **職責**: 宏觀分析產品在法規品目中的定位。
- **核心邏輯**: 分析上傳的文件內容（或貼上的文字），比對其申報的「品名」、「預期用途 (Intended Use)」與「分類碼」是否具備邏輯一致性。
- **推理特徵**: 著重於「策略層級」的合規判斷，防止品目申報錯誤。

#### 代理人二：技術審計員 (Agent 2: Technical Compliance Auditor)
- **職責**: 微觀數據偵錯。
- **核心邏輯**: 針對文件內部的性能指標進行交叉檢核。例如：檢查仿單 (D6) 上的滅菌期限是否與 D8 技術文件的測試結果一致。
- **推理特徵**: 高度數據驅動，尋找文字間的矛盾與數據偏差。

### 4.2 內容攝取與處理 (Data Ingestion)
- **多態輸入支援**: 
    - **文件上傳 (Upload Zone)**: 支援大體積 PDF 並可在本地「裁剪」。
    - **貼上區域 (Paste Zone)**: 針對法規公告、草稿、或零散的郵件回覆內容，用戶可直接貼入文字，點擊「Apply Paste」按鈕後，系統會在背景生成一個虛擬文本實體供代理人分析。
- **Consolidation (匯整機制)**: 兩位代理人的分析結果將在 UI 中併排顯示。系統提供「Consolidated Audit Output」，允許法規人員同步編輯兩個代理人的輸出結果，最終匯出為完整的內部審核筆記。

## 伍、 本地化自定義規則庫 (Rule Management Middleware)

### 5.1 mdpass 規則：路徑判定核心
- **定義**: 指定了不同情境下應必備的文件。
- **自定義能力**: 開放了 JSON 導出與導入接口。企業可以針對自身特定產品線（如專攻植入式血管支架）將特定的補充文件（如 D10 植入物追蹤清單）寫入規則庫中。

### 5.2 conflict 規則：智慧衝突提示
- **定義**: 定義了文件內容的正誤判定。
- **邏輯範例**: `(isSterile == true) AND (D8_Material_Biocomp == null)` -> 觸發警示：「警告：產品標示為無菌但技術資料中缺少生物相容性完整測試」。

## 陸、 安全性與隱私保護 (Security & Privacy)

在醫療產業中，文件即是機密。本系統實施「**端到端零存儲機制 (End-to-End Zero Persistence)**」：
- **無伺服器存儲**: 所有的上傳文件僅暫存於用戶瀏覽器的 RAM 中。一旦頁面重新載入，所有敏感數據自動焚毀。
- **前端裁剪技術**: 在本地處理 PDF，減少了與外部 AI 端點交換數據的體積，實現了極致的隱私最小化處理策略。

## 捌、 二十個深度追蹤與 Follow-up 問題 (Comprehensive Qs)

1.  **歐盟 MDR 映射機制**: 當企業依據歐盟 MDR 準備技術文件夾時，系統如何自動將其對應至 TFDA STED 的 D7 子目錄中？特別是關於 Post-Market Surveillance (PMS) 部分的差異判定？
2.  **臨床證據等同性判定**: Agent 如何判定臨床數據是否具備「台灣本土代表性」？
3.  **無菌級別與安定性關係**: 當產品被標記為無菌時，系統如何判定是否包含「包裝完整性試驗」？
4.  **軟體醫材 (SaMD) 漏洞檢核**: 針對含代碼醫材，Agent 是否能檢查網路安全說明是否包含「軟體物料清單 (SBOM)」？
5.  **QSD 全球有效期管理**: 系統如何處理同一製造廠名下多張 QSD 核准函的過期提示？
6.  **中國生產醫材產地標示規則**: 針對產地為大陸之產品，系統如何依據兩岸貿易規範檢查標籤上「分裝」或「製造」的字體標示？
7.  **標籤字體與圖示檢查**: Agent 能否識別 PDF 標籤中的「警告標語 (Warning Icons)」是否符合 CNS 1501 規範？
8.  **風險分析 (ISO 14971) 邏輯鏈**: 當 D9 提到的風險漏掉了 D7 的特定測試時，系統如何觸發不一致警報？
9.  **生物相容性豁免判定**: 針對「非接觸人體」之組件，如何自動生成法規豁免理由書 (Exemption Rationale)？
10. **複合產品 (Combination Products) 分類**: 系統如何協助 RA 判斷產品主體為「藥物」還是「醫材」？
11. **缺失函 (Deficiency Letter) 模擬**: 系統是否能透過歷史退件數據，模擬 TFDA 審查員的提問？
12. **電子仿單 (e-IFU) 實施判定**: 針對特定限制級醫材，系統如何導引用戶完成電子仿單的報備流程？
13. **類似品 (Similar) 目錄索引**: 系統是否能建立企業內部「曾核准產品」的數據指紋以便重複比對？
14. **Dual-Agent 意見歧異處理**: 當 Agent 1 認為合規但 Agent 2 認為有衝突時，系統如何記錄法規裁決決策樹？
15. **臨床試驗豁免路徑**: 針對已在美、歐、加、澳、日核准之醫材，系統如何自動勾選「免本土臨床」之路徑？
16. **授權書 (LoA) 權限查驗**: 系統如何檢查 LoA 中是否明確包含「副作用通報義務」之法律授權？
17. **體外診斷 (IVD) 特定邏輯**: 針對 IVD，判定引擎如何切換至檢驗清單 (Checklist for IVD)？
18. **UDI (唯一識別碼) 邏輯驗證**: 系統如何核對標籤上的 DI 與政府資料庫是否完全對應？
19. **Real-World Data (RWD) 採納策略**: 當臨床數據不足時，Agent 如何評估 RWD 的合規支撐強度？
20. **創新醫材綠色通道 (Breakthrough) 啟動**: 系統如何根據「極重度傷病」或「國內臨床急需」條件自動優化路徑？
